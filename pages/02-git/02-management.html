🐲 git 파일관리 🐲

🐪 개요

파일 관리하는 깃 명령어를 정리해봤다.

🐆 git pull

git pull의 정확한 명령어와 기능을 목록으로 정리했다.

git pull origin master
다른 사람이 PR을 통해서 코드를 업데이트 했거나 github를 통해서 커밋했을 때, 그 내용을 클라이언트로 내려받는 명령어
하나의 레포지토리를 다른 사람과 함께 작업하게 되면 작업자마다 변경된 사항이 다르게 되는데, 모든 작업자의 변경 사항이 적용된 파일을 다운받은 후에 작업해야 코드상의 혼선을 막을 수 있다. 이 때 사용하는 명령어이다.

🐖 git clone

git clone 정확한 명령어와 기능, git clone과 git pull의 차이를 목록으로 정리했다.

git clone 저장소 주소
클라이언트 상에 아무것도 없을 때, 서버의 프로젝트를 내려받는 명령어
빈 폴더 생성 후 bash에 두 명령어를 입력한다.
git clone와 git pull의 차이
작업물의 유무로 구분하여 사용한다.
사용하는 컴퓨터에 기존 작업물이 존재하면, git pull 명령어를 사용한다.
사용하는 컴퓨터에 기존 작업물이 없거나 주 작업장이 아니라면, git clone을 사용한다.

🌝 git commit 관리 🌝

🌚 개요

git commit을 확인하고 수정하는 명령어를 정리해봤다.

🌓 git 파일 상태

git 파일의 상태 종류를 목록으로 정리했다.

1 Unmodified
commit 후 변경사항이 없음을 의미한다.
2 Modified
파일 내용이 변경되었음을 의미한다.
git add 경로/파일명으로 스테이징한다.
3 Staged
git add를 하고 난 직후의 상태임을 의미한다.
commit 될 준비가 되었음을 의미하기도 한다.

🌔 git diff

git diff 정확한 명령어와 기능을 목록으로 정리했다.

git diff 경로/파일명
파일의 변경부분을 출력한다.

🌖 git checkout

git checkout의 기능을 목록으로 정리했다.

git checkout
수정을 잘못해서 파일을 원상태로 돌릴 때 사용하는 명령어
파일의 상태가 Modified에서 Unmodified로 변경된다.
git checkout하고나서 git diff 명령어를 입력하면 아무것도 뜨지않아야 한다.

🌘 git reset

파일을 잘못 스테이징 했거나(git add) 커밋을 잘못 입력했을 때, 되돌리는 명령어 git reset의 종류와 정확한 명령어, 기능을 목록으로 정리했다. 필요에 따라 명령어를 선택하여 입력한다.

1 git reset --soft 경로/파일명
해당 파일의 commit만 취소된다.
Unmodified에서 Staged로 되돌린다.
2 git reset (--mixed) 경로/파일명
해당 파일의 add와 commit이 취소된다.
git reset의 디폴트 값이다.
Unmodified에서 Modified로 되돌린다.
3 git reset --hard 경로/파일명
해당 파일의 파일 수정, add, commit이 취소된다.
Unmodified에서 Unmodified로 되돌린다.
변경사항도 모두 되돌리므로 사용에 주의를 기울여야 한다.

git reset의 HEAD 조작 명령어의 종류와 기능을 목록으로 정리했다.

git reset (옵션) HEAD
현재 HEAD의 모든 add와 commit이 취소된다.
git reset과 동일하다.
git reset (옵션) HEAD^1
바로 전 HEAD의 모든 add와 commit이 취소된다.

🌛 git push 취소하기

git push를 잘못했을 때 자신의 로컬 내용을 remote에 강제로 덮어쓰는 방법이다.
두 가지 주의할 점이 있다.

  • git push를 되돌리면, 해당 커밋 이후의 모든 정보는 사라진다.
  • 협업중이라면 동기화 문제가 발생하므로 팀원과 상의 한 후 진행한다.
가장 최근의 커밋을 취소한다.
git reset HEAD^
원하는 시점으로 워킹 디렉토리를 되돌린다.
깃허브에서 해당 디렉토리의 커밋 아이디를 복사한다.
git bash에 git reset 커밋 아이디 를 입력한다.
되돌아간 시점에서 add와 커밋 메세지를 입력한다.
git add 파일이름
git commit -m "커밋 메세지"
강제로 push한다.
git push -f origin 브랜치명

🌜 git commit 취소하기

git revert의 정확한 명령어와 기능, git reset과의 차이점을 목록으로 정리했다.

git reset --soft 커밋로그
커밋 메세지만을 수정할 수 있는 명령어이다. push하기 전에 실행한다.
git reset 하는 방법

git reset --soft 커밋로그 입력
git log로 커밋이 취소됐는지 확인 후에 git commit -m '커밋메세지'를 명령한다.
다시 git log를 명령하여 수정된 커밋 메세지를 확인한다.

git revert 커밋로그
push해서 서버에 업로드한 상태라면 커밋메세지를 수정할 수 없다. 대신, 해당 커밋의 바로 뒤에 변경한 메세지를 덧붙여서 올바른 커밋을 표현할 수 있다. 이 때 이용하는 명령어이다.
git revert 하는 방법

git revert 커밋로그 입력
화면이 바뀌면 Revert "수정할 커밋메세지"를 맨 윗줄에 쓴다.
:wq를 입력해서 창을 빠져나온 후, git log로 변경사항을 확인한다.

git revert와 git reset의 차이
push의 유무로 구분한다.
서버에 push하기 전에 커밋메세지를 수정하고싶다면, git reset을 사용한다.
서버에 push했는데 커밋메세지를 수정하고싶다면, git revert를 사용한다.

👽 git branch 관리 👽

💜 개요

git branch 생성, 이동, 삭제, 병합하는 명령어를 정리해봤다.

💛 branch를 생성해야 하는 이유

협업하기 위해 브랜치를 생성한다.
여러 팀원들이 각자 다른 파일을 작업할 땐 괜찮다.
문제는 '같은'파일을 작업하면 문제가 발생한다.
push할 때 현제 헤더상에서 commit 충돌이 일어나기 때문이다.
팀원마다 다른 브랜치를 형성하면 충돌 문제를 해결할 수 있다.
또한, commit관리에도 용이하다.

A 직원과 B 직원이 같은 파일을 작업하고 각각 상이한 작업 내용을 commit한다.
A 직원의 작업물에는 B 직원의 작업내용이 반영되지 않는다.
B 직원의 상황도 마찬가지다.
이 상태에서 push를 하면?
서버 입장에서는 현재 헤더에서 작업 내용과 commit이 제각각이니 push에러가 발생한다.
각자 다른 브랜치를 사용하여 헤더를 분리시켜야할 필요가 있다.
작업물을 반영하고 싶다면, push 한 뒤에 master 브랜치에 해당 브랜치를 병한다.

💞 branch 병합

각자의 branch에서 작업하던 것을 master 브랜치에 합쳐서 실제로 사용할 수 있는 코드로 만든다.
master 브랜치로 전환한 상태에서 명령한다. git merge 정확한 명령어와 기능을 목록으로 정리했다.

git merge 브랜치명
브랜치 내용을 병합하는 명령어
merge전에는 HEAD의 위치가 master에 있는지 확인한다.
브랜치 명에는 내용을 합칠 브랜치 이름을 입력해야 한다.
