🐲 git 파일관리 🐲

🐪 저장소 관리

🐆 git pull

git pull origin master
다른 사람이 PR을 통해서 코드를 업데이트 했거나 github를 통해서 커밋했을 때, 그 내용을 클라이언트로 내려받는 명령어
하나의 레포지토리를 다른 사람과 함께 작업하게 되면 작업자마다 변경된 사항이 다르게 되는데, 모든 작업자의 변경 사항이 적용된 파일을 다운받은 후에 작업해야 코드상의 혼선을 막을 수 있다. 이 때 사용하는 명령어이다.

🐖 git clone

git clone 저장소 주소
클라이언트 상에 아무것도 없을 때, 서버의 프로젝트를 내려받는 명령어
빈 폴더 생성 후 bash에 두 명령어를 입력한다.
git clone와 git pull의 차이
작업물의 유무로 구분하여 사용한다.
사용하는 컴퓨터에 기존 작업물이 존재하면, git pull 명령어를 사용한다.
사용하는 컴퓨터에 기존 작업물이 없거나 주 작업장이 아니라면, git clone을 사용한다.

🌝 git commit 관리

🌓 git status

Unmodified
commit 후 변경사항이 없음을 의미한다.
Modified
파일 내용이 변경되었음을 의미한다.
git add 경로/파일명으로 스테이징한다.
Staged
git add를 하고 난 직후의 상태임을 의미한다.
commit 될 준비가 되었음을 의미하기도 한다.

🌔 git diff

git diff 경로/파일명
파일의 변경부분을 출력

🌖 git checkout

git checkout
수정을 잘못해서 파일을 원상태로 돌릴 때 사용하는 명령어
파일의 상태가 Modified에서 Unmodified로 변경된다.
git checkout하고나서 git diff 명령어를 입력하면 아무것도 뜨지않아야 한다.

🌘 git reset

git reset 경로/파일명
파일을 잘못 스테이징 했거나(git add) 커밋을 잘못 입력했을 때, 되돌리는 명령어
git reset --soft 경로/파일명
해당 파일의 commit만 취소된다.
Unmodified에서 Staged로 되돌린다.
git reset --mixed 경로/파일명
해당 파일의 add와 commit이 취소된다.
git reset의 디폴트 값이다.
Unmodified에서 Modified로 되돌린다.
git reset --hard 경로/파일명
해당 파일의 파일 수정, add, commit이 취소된다.
Unmodified에서 Unmodified로 되돌린다.
변경사항도 모두 되돌리므로 사용에 주의를 기울여야 한다.

👽 git reset head

git reset (옵션) HEAD
현재 HEAD의 모든 add와 commit이 취소된다.
git reset과 동일하다.
git reset (옵션) HEAD^1
바로 전 HEAD의 모든 add와 commit이 취소된다.

🌛 git push 취소하기

git push를 잘못했을 때 자신의 로컬 내용을 remote에 강제로 덮어쓰는 방법이다.
두 가지 주의할 점이 있다.

  • git push를 되돌리면, 해당 커밋 이후의 모든 정보는 사라진다.
  • 협업중이라면 동기화 문제가 발생하므로 팀원과 상의 한 후 진행한다.
git reset HEAD^
가장 최근의 커밋 취소
git reset 커밋 아이디
깃허브에서 원하는 시점의 커밋 아이디를 복사
원하는 시점으로 디렉토리를 되돌린다.
되돌아간 시점에서 add와 커밋
git add 파일이름
git commit -m "커밋 메세지"
git push -f origin 브랜치명
강제 push

🌜 git commit 취소하기

git reset --soft 커밋로그
커밋 메세지만을 수정할 수 있는 명령어
push하기 전에 실행
git revert 커밋로그
push해서 서버에 업로드한 상태라면 커밋메세지를 수정할 수 없다. 대신, 해당 커밋의 바로 뒤에 변경한 메세지를 덧붙여서 올바른 커밋을 표현할 수 있다. 이 때 이용하는 명령어이다.
수정할 메세지는 revert 명령 후, 바뀐 화면 맨 윗줄에 입력
:wq를 입력해서 창 종료 후, git log로 변경사항을 확인
git revert와 git reset의 차이
push의 유무로 구분한다.
서버에 push하기 전: git reset 사용
서버에 push한 후: git revert 사용

👽 git branch 관리

💛 branch를 생성해야 하는 이유

협업하기 위해 브랜치를 생성한다.

다른 사람과 동일한 파일을 작업하고 push할 때, commit 충돌이 일어나기 때문이다.

A 직원과 B 직원이 하나의 브랜치로 동일한 파일을 작업하고 commit하면, 다른 사람이 작업한 내용은 파일에 반영되지 않는다.
이 상태에서 push를 하면,
서버 입장에서는 현재 헤더에서의 작업 내용과 commit이 제각각이니 push에러가 발생한다.
브랜치를 분리하면, commit 충돌을 예방할 수 있다.

💙 branch 생성

git branch 브랜치명
새로운 브랜치 생성하는 명령어
git branch
브랜치 목록 확인
현재 HEAD가 있는 브랜치는 *표시가 되어있다.

❤️ branch 전환

git checkout 브랜치명

💞 branch 병합

(참고) 각자의 branch에서 작업하던 것을 master 브랜치에 합쳐서 실제로 사용할 수 있는 코드로 만든다.

(참고) master 브랜치로 전환한 상태에서 명령한다.

git merge 브랜치명
브랜치 내용을 병합하는 명령어
브랜치 명에는 내용을 합칠 브랜치 이름을 입력해야 한다.
